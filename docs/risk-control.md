# 账号共享控制系统配置教程

本指南介绍如何在总控后台与子节点环境配置防共享参数，以及如何调整阈值来控制警告与封禁行为。

---
## 注意事项
* 该功能在正式运营时，开启检测后一定不要先开启封禁，可以经过长时间测试后 得到一个正确的适合自己运营的阈值再把封禁启用

## 1. 总控侧阈值与控制策略配置

请登录 SUI-Ops 总控后台，进入 系统设置 -> 账号共享控制 页面进行策略调整。

### 1.1 基础风控开关（总控顶层）
* 启用风险控制：开启后系统会自动监控用户行为并进行风险评估（对应总控全局开关）。
* 自动禁严严重异常用户：开启后系统会自动封禁严重异常的用户（对应 hard block 自动执行开关）。

### 1.2 节点行为参数
* 用户状态过期时间(分钟)：多久没有上报行为，就删除该用户风控状态。推荐值 5 分钟。
* 活跃节点判断窗口(秒)：在此时间内有上报即视为活跃节点。默认 10 秒。
* 快速切换判定阈值(秒)：节点切换间隔小于此值视为快速切换。默认 30 秒.
* 频率统计窗口(秒)：用于统计切换频率的时间跨度。默认 60 秒。
* 频率统计阈值(次数)：在频率统计窗口内切换次数达到此值，额外加分。默认 3 次。

### 1.3 风险评分阈值与衰减
* 警告阈值(soft block)：达到此分数后系统记录用户警告分数（注：此项仅作后台风控评级，不会通知或影响客户端）。默认 60 分。
* 封禁阈值(hard block)：达到此分数后全自动封禁用户凭证。默认 90 分。
* 风险衰减系数：旧行为影响逐渐降低，防止用户早期异常导致永久高风险。推荐值 0.6 - 0.85，默认 0.75。

### 1.4 节点行为评分加分项
* 双节点活跃加分：同时活跃两个节点时的额外得分。默认 5.0 分。
* 三节点及以上活跃叠加分：同时活跃三个及以上节点时的额外得分。默认 10.0 分。
* 快速切换单次加分：快速切换节点的单次额外得分。默认 5.0 分。
* 频率切换额外加分：频繁切换节点的额外得分。默认 8.0 分。

## 2. 子节点环境配置 (nodeRisk.json)

若需调整单个 VPS 节点的底层连接与 IP 统计权重，请直接使用 root 用户登录该节点 VPS，修改环境目录下的 nodeRisk.json 配置文件。该配置与总控端评分联动。

### 节点参数调整说明
* windowMs: 行为统计的时间窗口（单位：毫秒）。默认 10000（10秒）。
* ipWarn 与 ipLimit: 分别为开始计分线与加倍计分线。默认 3 和 6。
* connWarn 与 connLimit: 分别为连接数警告线（默认10）与连接数限制线（默认25）。
* burstWeight: 突发权重（默认25）。当连接数≥8且独立IP数≥3时直接叠加。
* finalScale: 节点侧最终分数的全局缩放系数，默认 1.0。
* protocolWeights: 协议插件字典。支持针对不同协议单独调整风险权重系数。例如发现 hysteria2（默认1.5）或 tuic（默认1.7）频繁因为高并发导致误报，可将其数值下调至 1.0。

## 3. 特殊场景优化的最佳实操方案

在实际运营中，严禁为了迁就个别多设备软路由大户而盲目调低总控侧的全局加分项（否则会导致正常用户的防共享机制全面失效）。请遵循以下生产规范进行分类处理：

* **方案 A：软路由多链路/大客户多设备并发场景（独立 VPS 物理隔离方案）**
  - **规范建议**：应单独创建或划拨专用的 VPS 服务器作为软路由/大客户专属节点，并从分配逻辑上进行套餐隔离。
  - **物理隔离实操**：直接登录该独立 VPS 节点，修改环境目录下的 nodeRisk.json 文件，将首行的 enable 修改为 false。此时，该节点将彻底关闭风控检测且不向总控提交任何风险报告，从根源上解决策略分流引发的串线警告或封禁问题。
* **方案 B：缓解正常用户因手机高频切换基站带来的偶尔警告**
  - **实操**：如果发现普通用户由于移动 network 漂移偶尔堆积历史分数，请登录总控后台，将 风险衰减系数 适当调低（例如由 0.75 下调至 0.6）。这能让偶尔的突发并发或网络漂移分数在进入下一个检测周期时，以更快的速度无感衰减淡出。

## 4. 运营常见问题 (FAQ)

**Q1：用户频繁切换节点会触发封禁吗？**
* **解答**：不会。单次切换节点仅给总控追加 5 分，即使在 60 秒内连续切了 3 次也只额外追加 8 分，衰减系数会在每个周期无感消散，累计最终风险分远低于 90 分的封禁线。

**Q2：手机移动网络 IP 频繁变动会触发封禁吗？**
* **解答**：不会。系统底层逻辑在独立 IP 数大于等于 3 时才开始计分，移动运营商网络基站切换导致的正常 IP 变动不会触发恶意加分惩罚。

**Q3：同一个账号多个设备同时使用会触发封禁吗？**
* **解答**：如果多设备连接同一个节点出口（IP 唯一），由于只在单节点产生连接数上升，属于安全范围；但如果多设备持续连接不同的节点（如分流软路由、多端异地同时合租），系统在每个检测周期都会检测到多节点并发并持续加分。通过系统的 EMA（指数移动平均）算法迭代，这种**持续的并发加分会不断堆叠累积**，最终必定会触发 90 分的封禁线。因此，针对软路由或大户场景，必须按照 **方案 A** 进行专用节点物理隔离。

**Q4：如果某个套餐或大代理被误封了，该怎么调整？**
* **解答**：请首先通过 **方案 A** 确认是否需要为其分配关闭了检测的独立 VPS 节点。若属于普遍行为，可在后台将 封禁阈值 (scoreBlock) 调高到 100 或 120，或者将 风险衰减系数 下调至 0.6 以加快分数的清理速度。

**Q5：如果普通节点（正常用户核心池）发生误封该怎么办？**
* **解答**：普通节点严禁直接关闭风控开关，否则会导致防共享机制全线失灵。如果普通节点出现误封，运营者应根据波及范围，选择以下两条路径之一进行详细调优：

  * **路径一：全网统一放宽（总控侧调整）**
    如果是跨节点的普遍性误封，说明当前的全局封禁线卡得太死。请登录总控后台进入 `账号共享控制` 页面：
    1. **提高容错上限**：将 **封禁阈值 (scoreBlock)** 统一上调 阈值 分，拉开警告线与封禁线之间的缓冲空间。
    2. **加速历史淡出**：将 **风险衰减系数 (decay)** 调低（例如由 `0.75` 降至 `0.6`）。这样用户偶尔因临时网络波动产生的历史高分，会在接下来的检测周期中成倍数地迅速衰减清零。

  * **路径二：单节点定向降权（子节点侧调整）**
    如果仅是某个特定地区、或使用了某种特定高并发协议的普通节点频繁发生误封，说明是单节点侧的采集参数与业务不匹配。请 root 登录该异常节点 VPS 修改 `nodeRisk.json`：
    1. **放宽触发基准线**：将 `ipWarn`（默认3）和 `connWarn`（默认10）分别适当调高（例如改为 `4` 和 `15`），让子节点侧的初始计分门槛更宽松。
    2. **针对特定协议做钝化处理**：如果是 Hysteria2 或 Tuic 等高频并发协议在该节点上容易爆表，可在 `protocolWeights` 字典中，将对应协议的权重由 `1.5` / `1.7` 统一拉低到 `1.0`，使其跟标准 TCP 协议的加分判定完全保持一致。

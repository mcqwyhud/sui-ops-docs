# API 接入与认证说明

> 本文档介绍外接系统如何通过 SUI-Ops 总控中心的 API 管理员账号获取及维护安全令牌（Token）。

---

## 1. 接入前准备（创建最小权限管理员）

为了确保总控数据安全，**严禁使用高权限的日常对接 API**。外接系统（如发卡网、用户前端）必须使用独立的 API 专属管理员账户。

### 安全操作规范：
1. 登录 SUI-Ops 后台，进入 **管理员管理 -> 添加管理员**。
2. 设定专用用户名（如 `api_client_main`）。
3. **密码安全要求**：密码建议复杂度越高越好，且包含大小写字母、数字与特殊符号（推荐直接使用密码生成器生成随机乱码）。

---

## 2. 认证接口规范

* **接口基地址 (Base URL)**：`https://<你的总控地址>/api/auth`
* **Token 有效期**：`5分钟`（过期后该 Token 自动进入不可用状态）

### 2.1 登录获取令牌

* **请求请求**：`POST /login`
* **Content-Type**：`application/json`

**请求参数示例**：
```json
{
  "username": "api_client_main",
  "password": "密码"
}
```

**成功响应示例**：
```JSON
{
    "code": 200,
    "msg": "success",
    "data": {
        "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.ey..."
    }
}
```

- 安全防护提示：总控内置防爆破锁定机制。若连续输错密码达到系统限制，该 API 账户将被全自动锁定。锁定期间即使密码正确也会返回 400 错误。

## 3. 令牌维护与平滑刷新
- 为了不中断您的后台自动化调用，必须在 5分钟 的生存窗口期内，利用老 Token 去调换新 Token。

### 3.1 刷新令牌 (Refresh Token)
- 请求请求：POST /refresh-tokenHeaders 
- 传参规范：必须在 HTTP 头中携带老令牌：Authorization: Bearer <当前尚未过期的老Token>
- 请求响应示例：
```JSON
{
   "code": 200,
   "msg": "success",
   "data": {
       "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.new_token_ey..."
   }
}
```
### 3.2 异常拦截返回码

在调用刷新接口（`/refresh-token`）或后续其他业务接口时，如果遇到以下错误返回结构，外接系统（如发卡网、用户前端、自动化脚本）应立即触发**重新登录（Login）**机制，重新走登录接口获取新 Token：

| 业务错误提示 (`msg`) | 触发原因 | 开发者外接处理策略 |
| :--- | :--- | :--- |
| **Token不能为空**<br>或 **Token无效** | 传入的 `Authorization` 头格式错误、未加 `Bearer ` 前缀，或者 Token 被中途篡改导致 JWT 验签失败。 | 检查外接系统本地的 Header 拼接与传输逻辑。 |
| **Token不在刷新窗口内，无法刷新** | 该 Token 的生存期已完全消耗完毕（超过 5 分钟限制），彻底滑出允许刷新的临界窗口。 | 立即启动 `2.1` 登录流程，通过用户名和密码重新获取初始 Token。 |
| **Token已失效** | 该 Token 在总控服务端的缓存（`TokenCacheService`）中已被主动擦除，通常是由于被其他同账号调用挤掉、或已触发登出。 | 清除本地旧 Token 缓存，重新发起登录。 |
| **用户已被禁用**<br>或 **账户已被锁定** | 该专用 API 管理员账户在 SUI-Ops 总控后台被运维人员人工封禁，或者由于密码输错次数过多触发了系统的防爆破锁定保护。 | 外接脚本必须立即停止高频调用，防止引发连带报错；需人工前往总控后台检查该 API 账号的状态。 |
